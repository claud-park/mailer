# label-crud — DECISIONS

> 2026-07-16. D1은 대화형 확정, D2~D4는 구현 세부 추천안.

## D1. 라벨 삭제는 확인 다이얼로그 필수 — 사용자 확정
- **선택지**: (a) 확인 다이얼로그, (b) 즉시 삭제(undo 토스트로 보완).
- **선택**: (a).
- **이유**: 라벨 삭제는 "그 라벨이 붙은 모든 메일에서 제거"라는 전역적 파괴 행위이고, Gmail 웹 자신도 확인을 요구한다. undo-toast(별도 feature)가 존재하더라도, 대상이 수십~수백 통일 수 있는 라벨 삭제는 "실행 후 후회하면 되돌리기"보다 "실행 전에 한 번 더 확인"이 적절한 리스크 등급.

## D2. 이름만 입력, 색상은 Gmail 자동 할당 — 추천안
- **선택지**: (a) 이름만(색상 자동), (b) 색상 선택 UI 동반.
- **선택**: (a).
- **이유**: PRD가 요청한 건 "추가·삭제"뿐 — 색상 선택 UI는 스코프 확장이고, Gmail도 라벨 생성 시 기본 팔레트에서 자동 배정하므로 이후 필요하면 Gmail 웹에서 바꿀 수 있다(v1 범위 밖, YAGNI).

## D3. 삭제 아이콘은 hover-reveal, 상시 노출 아님 — 추천안
- **선택지**: (a) hover 시에만 노출(사이드바가 파괴 아이콘으로 어수선해지지 않음), (b) 항상 노출.
- **선택**: (a).
- **이유**: 사이드바는 이 앱의 핵심 내비게이션 표면이라 평상시엔 깔끔해야 하고, 삭제는 의도적으로 그 행에 마우스를 올려야만 접근 가능한 정도의 약한 마찰이 적절하다(select-all-in-view 등 기존 feature들도 파괴적 액션엔 유사한 의도적 마찰을 둠).

## D4. Mock provider의 라벨 삭제는 스레드 labelIds도 함께 정리 — 추천안
- **컨텍스트**: Real Gmail은 라벨 삭제 시 그 라벨을 가진 모든 메시지에서 자동으로 제거된다(서버 측 동작). Mock은 이를 흉내내야 데모 모드에서 TC-LBL-B3(칩이 실제로 사라지는지)를 검증할 수 있다.
- **선택**: `MockGmailProvider.deleteLabel(labelId)`가 `this.labels`에서 제거함과 동시에 `this.threads`의 모든 `summary.labelIds`/`detail.labelIds`에서 해당 id를 제거.
- **이유**: 그렇지 않으면 데모에서 삭제된 라벨의 칩이 스레드 목록에 유령처럼 남아 실계정과 다른 시맨틱이 된다(이 프로젝트가 반복적으로 강조하는 "데모=실계정 패리티" 원칙).
