# select-all-in-view — DECISIONS

> Goal 4 산출물. 사용자 브레인스토밍 질문에 60s 무응답 → 추천안 채택. 아래 D1~D5는 전부 **⚠️ 사용자 미확인** — 복귀 시 확인 필요(F1 D1~D3, F3 D3/D13 선례와 동일 관례).

### D1. 트리거 = ⌘A, useKeyboard 소유, 타이핑/모달 중엔 미발화 (⚠️ 미확인 추천안)
- **이유**: 코드 확인 결과 ⌘A는 kbar 단일키 목록에도, useKeyboard 모디파이어 콤보 목록에도 없는 완전한 공백. 모디파이어 조합이라 CLAUDE.md 단축키 소유권 규약상 useKeyboard 소유가 맞음. isTyping(e.target) 검사 뒤에 배치해 검색창/컴포즈 등에서는 브라우저 네이티브 텍스트 전체선택이 그대로 동작하게 함(사용자 기대와 충돌 방지).

### D2. 일괄 액션 = 기존 단축키(e/#/I/U/l/b) 전부 재사용, 신규 UI 액션 없음 (⚠️ 미확인 추천안)
- **이유**: 브레인스토밍 질문에서 제시한 두 옵션(전부 지원 vs 아카이브만) 중 "전부 지원"을 사용자 응답 부재로 기본 채택 — Superhuman/Gmail 벤치마킹상 아카이브만 되는 select-all은 실용성이 낮음. 기존 단축키 체계와 자연스럽게 결합되어 학습비용이 0에 가까움.
- **범위 결정(YAGNI)**: 개별 행 체크박스/⌘클릭/Shift+클릭 다중 선택은 v1에서 배제 — 사용자가 문자 그대로 요청한 것은 "select all"이지 범용 멀티셀렉트가 아님. 필요해지면 후속 확장.

### D3. 개별 토스트 억제 + 집계 토스트 1건 (검증됨, 자유도 낮음)
- **이유**: 기존 `toast: string | null`은 단일 상태라 10건을 순차 처리하면 마지막 메시지만 남는 것을 코드로 확인. archiveThread 등에 `silent` 옵션을 추가해 bulk 호출 시 개별 토스트를 억제하고, bulk 래퍼가 완료 후 "N개 아카이브됨" 1건만 표시.

### D4. 액션 후 선택 자동 해제 (⚠️ 미확인 추천안)
- **대안**: 액션 후에도 선택 유지(연속으로 다른 액션 체이닝 가능) — 기각. 아카이브/트래시는 대상이 뷰에서 사라지므로 "선택 유지"가 오히려 혼란(사라진 스레드를 계속 선택 중이라고 오인). "일괄 처리 = 원샷" 멘탈 모델이 가장 예측 가능해 자동 해제를 기본으로.

### D5. 시각 표시 = `bg-accent/10` 행 강조 + 언리드 도트 자리 체크 표시 + 상단 슬림 배너 (⚠️ 미확인 추천안)
- **이유**: 기존 단일 커서 강조(`bg-bg-subtle`)와 겹치지 않는 별도 톤 필요. 도트 자리를 재활용하면 레이아웃 변경 최소화. 배너는 FollowupBanner(스레드 상단 배너) 톤을 리스트 상단으로 재사용 — 신규 시각 언어 도입 최소화(미니멀리즘 유지).

## 실행 안전성 (검증됨, 결정 아님)
- 다건 순차 호출 시 각 호출의 동기 optimistic 갱신이 `await` 이전에 완료되므로 레이스 없음(코드 확인). F6 per-thread FIFO 장벽은 스레드별이라 여러 스레드 동시 처리에 특별한 장벽 로직 불필요. F4 계측은 기존 액션 id를 그대로 재사용(신규 카테고리 없음, D2 폴로우).
