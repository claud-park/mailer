# remote-image-prefetch — DECISIONS

> Goal 4 산출물. 브레인스토밍 확정([설계 스펙](../../superpowers/specs/2026-07-19-remote-image-prefetch-design.md))에서 사용자 확정 항목(D1~D5)과, 스펙 구현 세부를 다루는 추천안(D6~D9)으로 구성.

## D1. 목표 수준 = 로컬 프리페치(진짜 익명 프록시 미채택) — **사용자 확정**
- **컨텍스트**: Superhuman/Gmail 웹의 무클릭 이미지 로드는 공유 IP로 이미지를 릴레이하는 자체 호스팅 프록시 백엔드에서 나온다. ZenMail은 백엔드가 없는 로컬 Electron 앱이라 완전히 동일한 익명성(발신자 입장에서 특정 사용자를 식별 못 함)은 재현할 수 없다는 사실을 사용자에게 먼저 설명.
- **선택지**: (a) 로컬 프리페치 — 실제 IP는 노출되지만 열람 시점과 요청 시점은 분리, 추가 인프라 불요, (b) 진짜 익명 프록시 — 호스팅 백엔드 신설(공유 IP), 인프라 구축·운영 비용 발생, 앱 설계 의도(미니멀 로컬 앱) 재검토 필요, (c) 게이트만 제거(프리페치도 프록시도 없이 renderer가 직접 원격 fetch) — 구현은 가장 간단하지만 시점 분리 이득이 전혀 없어 원래 게이트가 막던 문제를 그대로 재도입.
- **선택**: (a) 로컬 프리페치.
- **이유**: (b)는 이 프로젝트의 규모(개인/소수용 미니멀 Electron 앱)에 비해 인프라 비용·운영 부담이 과도하고, 별도의 백엔드 서비스 프로젝트가 되어 아키텍처 성격 자체가 바뀐다. (c)는 구현은 쉽지만 애초에 게이트를 만들었던 이유(트래킹 픽셀 방어)를 완전히 무력화한다. (a)는 실제 IP 노출이라는 한계는 있지만 Gmail 웹 기본값과 동등한 수준의 트레이드오프이며, 최소한 "정확히 언제 열었는지"는 감춘다 — 추가 인프라 없이 로컬 아키텍처 안에서 구현 가능하다.

## D2. 프리페치 범위 = 동기화된 전체 받은편지(Gmail 기본값과 동일) — **사용자 확정**
- **컨텍스트**: 어디까지 프리페치할지 — 받은편지함 전체를 동기화 시점마다 미리 받을지, 아니면 스레드 리스트에 실제로 노출된 것만, 혹은 사실상 열람 직전에만 가져올지. 범위가 넓을수록 시점 분리 효과는 커지지만, 읽지 않는 메일(스팸 포함)에도 트래킹 픽셀이 100% 발화된다는 트레이드오프가 있음을 먼저 명시.
- **선택지**: (a) 동기화된 전체 받은편지 — 안 읽은 메일도 트래킹 픽셀 100% 발화되지만 시점 분리 효과 최대, Gmail 웹 기본값과 동일 수준, (b) 스레드 리스트에 노출된 것만 — 발화 범위는 줄지만 목록에 뜨는 순간 이미 발화되므로 시점 분리 효과가 약함, (c) 열람 직전만 — 구현은 제일 간단하지만 시점 분리 이득이 거의 없음(D1에서 이미 기각한 (c)와 사실상 동일).
- **선택**: (a) 동기화된 전체 받은편지.
- **이유**: 사용자가 "안 읽은 메일도 트래킹 픽셀이 항상 발화된다"는 트레이드오프를 명시적으로 인지한 뒤 채택 — 이는 Gmail 웹이 오랫동안 기본값으로 채택해온 동작과 동일한 수준이라 업계 기준에서 벗어나지 않는다. (b)는 발화 시점이 "목록에 뜬 순간"이라 여전히 사용자의 실시간 브라우징과 상관관계가 생겨 시점 분리 취지를 절반만 달성한다.

## D3. 백필 범위 = 신규 메일부터만(과거 메일 소급 없음) — **사용자 확정**
- **컨텍스트**: 기능을 켜는 시점에 이미 동기화된 과거 메일(수년치 가능)까지 소급 프리페치할지, 그 이후 도착하는 메일부터만 적용할지.
- **선택지**: (a) 신규 메일부터만 — 구현 간단, 대역폭/시간 예측 가능, (b) 최근 N일 백필 + 신규 — 초기 대역폭/시간 부담 발생, 구현 복잡도 증가(백필 진행률/중단/재개 등 신규 상태 관리 필요).
- **선택**: (a) 신규 메일부터만.
- **이유**: 과거 메일은 열람 시 캐시 miss → 즉석 fetch로 채워지므로(D1 설계상 자연스러운 fallback), 결과적으로 한 번이라도 연 과거 메일은 그다음부터 즉시 로드된다. 수년치를 무조건 백필하는 건 v1 범위에서 얻는 이득 대비 구현·운영 비용(진행률 UI, 취소, 재개)이 크다 — YAGNI.

## D4. 토글 위치 = Command Palette — **사용자 확정**
- **컨텍스트**: 새 on/off 토글을 어디에 둘지. 현재 앱에는 범용 Settings 패널이 없고, theme 토글이 유일한 선례로 Command Palette에 달려 있음.
- **선택지**: (a) Command Palette에 추가(theme 토글과 동일 패턴), (b) 이 기회에 최소 Settings 패널 신설.
- **선택**: (a) Command Palette.
- **이유**: (b)는 토글 하나 때문에 새 UI 서페이스(라우팅, 레이아웃, 향후 다른 설정들의 정착지 결정)를 만들어야 해 이번 feature 범위를 벗어난다. 기존 theme 토글이 이미 동일한 패턴(전역 설정 + Command Palette 액션)으로 검증돼 있어 그대로 복제하는 것이 가장 낮은 리스크·구현 비용이다.

## D5. 토글 기본값 = on(설정에서 off 가능) — **사용자 확정**
- **컨텍스트**: 기본값을 "자동 로드"로 바꾸는 것 자체가 프라이버시 기본값의 전환이다 — 토글을 두되 기본 on으로 할지, 아예 게이트 자체를 완전히 제거(토글 없음)할지.
- **선택지**: (a) 설정에 on/off 토글, 기본값 on, (b) 게이트 완전 제거(토글 없이 항상 자동 로드).
- **선택**: (a) 토글 + 기본값 on.
- **이유**: (b)는 구현이 더 단순하지만 프라이버시를 원치 않는 사용자에게 후퇴할 방법을 주지 않는다. 기본값을 켜짐으로 바꾸는 것(Gmail 웹과 동일 기조, D2)과 "완전히 선택지를 없애는 것"은 별개 문제 — 토글을 유지하면 이번 feature가 기존 클릭-게이트 사용자에게도 무손실 상위호환이 된다(FR16).

## D6. 프리페치 트리거 지점 = 기존 unread-count 순증 훅에 얹되, bodyHtml 확보용 `getThread` 호출은 신규 추가(새 폴링 루프는 신설 안 함) — 추천안
- **컨텍스트**: 프리페치를 언제 실행할지 조사하며 확인한 사실 정정 — new-mail-alerts D8 훅(`snooze.ts` unread count 순증 시)이 이미 부르는 `listThreads({q: NEW_MAIL_QUERY})`는 **`ThreadSummary[]`만 반환하고 `bodyHtml`을 포함하지 않는다**(알림 배너의 발신자/제목 표시에만 필요했기 때문). 애초 설계 스펙 초안은 "이미 확보된 bodyHtml 재사용, 별도 fetch 없음"이라고 적었으나 이는 틀렸다 — 이미지 URL을 추출하려면 신규 스레드마다 `provider.getThread(threadId)` 상세조회를 **새로 추가**해야 한다.
- **선택지**: (a) unread count 순증 트리거는 그대로 재사용하되, `diffNewUnread`가 골라낸 신규 스레드에 한해 `getThread()`를 추가 호출(신규 API 콜, 다만 기존 알림 fetch와 동일하게 "카운트가 오를 때만"이라는 자연스러운 상한이 있음), (b) 별도 프리페치 전용 폴링 타이머 신설(어떤 스레드가 신규인지부터 다시 판별해야 해 로직 중복), (c) bodyHtml을 얻을 수 있는 유일한 기존 경로(사용자가 실제로 스레드를 여는 `mail:fetch-thread`)에만 의존(=D1의 즉석 fetch fallback과 동일해져 "사전" 프리페치가 사실상 사라짐).
- **선택**: (a) 기존 트리거 재사용 + `getThread` 신규 추가.
- **이유**: (c)는 시점 분리 효과가 사실상 없어 D1/D2에서 채택한 "로컬 프리페치"의 존재 이유를 무력화한다. (b)는 "이 스레드가 새로 온 것인지"를 판별하는 로직(`diffNewUnread`+`lastKnownUnreadIds`)을 통째로 복제해야 해 두 군데서 어긋날 위험만 늘린다. (a)는 API 호출이 늘어난다는 점을 솔직히 인정하되, 그 호출 빈도가 "계정의 unread count가 실제로 오른 만큼"으로 자연스럽게 상한선이 있어(스팸이 쏟아지는 계정이 아닌 한 분당 소수 건) F6/new-mail-alerts D7이 지켜온 "API 비용 원칙"을 크게 벗어나지 않는다. PRD FR7/FR8은 이 정정을 반영한다.

## D7. iframe CSP `img-src`는 영구적으로 `data:`만 허용(https:/http: 스킴을 다시는 노출하지 않음) — 추천안
- **컨텍스트**: 기존 `allowImages` 게이트는 통과 시 iframe 메타 CSP의 `img-src`에 `https: http:`를 추가해 renderer(iframe)가 원격 이미지를 직접 fetch하도록 했다. 이 feature는 이미지를 main 프로세스가 대신 가져와 data URI로 넘겨주므로, iframe이 원격 스킴을 직접 다룰 이유가 사라진다.
- **선택지**: (a) CSP `img-src`를 `data:`로 고정하고 모든 원격 이미지를 data URI 치환으로만 표시, (b) 기존처럼 게이트 통과 시 `https: http:`를 CSP에 추가하는 경로를 유지(캐시 miss 시 fallback으로).
- **선택**: (a) `data:` 고정.
- **이유**: (b)를 유지하면 캐시 miss 시 iframe이 직접 원격 URL에 접속하는 경로가 부활해, main 프로세스의 SSRF 가드(D9)를 완전히 우회하는 별도 네트워크 경로가 생긴다(iframe은 사용자의 실제 브라우저 네트워크 스택을 그대로 쓰므로 사설 IP 접근도 막을 수 없음). (a)는 모든 원격 이미지 요청이 예외 없이 main 프로세스의 `getCachedOrFetch`(SSRF 가드 포함)를 거치도록 구조적으로 강제한다 — 기존보다 오히려 공격 표면이 줄어든다.

## D8. 캐시 바이트는 디스크 파일, 메타데이터만 sqlite(attachments D5와 정반대 — 이유가 다름) — 추천안
- **컨텍스트**: attachments feature의 D5는 "첨부 바이트는 sqlite에 저장하지 않고 매 요청 fresh fetch"였다. 이 feature는 반대로 "캐시가 곧 기능의 목적"이므로 동일한 결론을 낼 수 없다 — 다만 바이트를 sqlite BLOB으로 넣을지 별도 디스크 파일로 둘지는 여전히 선택지.
- **선택지**: (a) 바이트는 계정별 디렉터리에 파일(`<sha256(url)>`)로, sqlite에는 조회/prune용 메타데이터만, (b) sqlite BLOB 컬럼에 바이트까지 저장.
- **선택**: (a) 파일 + 메타데이터 분리.
- **이유**: attachments D5의 "바이트 캐시 제외" 판단 근거(sqlite 비대화 방지)는 이 feature에도 여전히 유효하다 — 다만 이 feature는 캐시 자체가 목적이므로 "sqlite에 안 넣는다"가 아니라 "sqlite에는 메타데이터만, 바이트는 파일시스템"으로 절충한다. WAL 모드 sqlite에 대용량 BLOB을 수백~수천 건 채우면 vacuum·백업·WAL 체크포인트 비용이 커지고, 파일시스템은 애초에 이런 워크로드(불변 바이트를 키로 조회)에 최적화돼 있다. `download.ts`의 안전한 파일쓰기 패턴을 재사용할 수 있다는 부수 이점도 있다.

## D9. SSRF 가드 = 사설/루프백/링크-로컬 IP 차단 + 리다이렉트 매 hop 재검증 + 크기/타입/시간 제한 — 추천안
- **컨텍스트**: 기존엔 사용자의 명시적 클릭이 곧 "이 URL에 접속해도 좋다"는 동의였다. 이 feature는 도착하는 모든 메일에 대해 자동으로 발화되므로, 피싱 메일이 사설 IP·클라우드 메타데이터 엔드포인트를 가리키게 해 SSRF를 시도할 공격 표면이 새로 생긴다.
- **선택지**: (a) 스킴 제한(`http`/`https`만) + 사설/루프백/링크-로컬 대역 차단 + 리다이렉트 매 hop 재검증 + `Content-Type: image/*` 강제 + 5MB/8초 제한, (b) 스킴만 제한하고 IP 대역 검사는 생략(구현 단순화), (c) 사용자에게 "자동 로드를 켜면 SSRF 위험이 있을 수 있다"고 경고만 하고 가드 없이 출하.
- **선택**: (a) 전체 가드.
- **이유**: (b)/(c)는 방어를 사용자 인지나 신뢰에 의존시키는데, 이 feature의 핵심 전제 자체가 "사용자가 인지·클릭하지 않아도 자동으로 일어난다"는 것이라 모순된다. 첫 URL만 검사하고 리다이렉트로 우회되는 경로를 막으려면 매 hop 재검증이 필수다(단순 스킴 검사만으로는 `https://trusted.example/redirect?to=http://169.254.169.254/...` 같은 우회를 못 막음). 크기/타입/시간 제한은 SSRF와 별개로 디스크 고갈·응답 지연을 통한 자원 소모 공격을 막기 위한 최소 방어선이다.
