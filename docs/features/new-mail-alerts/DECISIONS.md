# new-mail-alerts — DECISIONS

> 2026-07-19. 브레인스토밍(대화형 MCQ 6문항) 산출물. D1~D6는 사용자 확정, D7~는 구현 세부를 다루는 추천안.

## D1. 배지·알림 범위 = 전체 계정 합산 — 사용자 확정
- **선택지**: (a) 활성/비활성 무관 전체 계정 unread 합산, (b) 활성 계정만.
- **선택**: (a).
- **이유**: 멀티 계정 구조상 비활성 계정도 60s 데몬으로 unreadCount가 갱신되고 있는데, 배지가 활성 계정만 반영하면 백그라운드 계정 새 메일을 영영 놓친다 — Mail.app/Superhuman 관례와도 일치.

## D2. "새 메일" 기준 = Inbox(+Starred) unread 증가 — 사용자 확정
- **선택지**: (a) Inbox∪Starred(archive/trash/spam/스누즈 제외) unread 스레드 집합 증가, (b) 순수 INBOX 라벨 unread 증가만.
- **선택**: (a).
- **이유**: `starred-view`가 확정한 "Starred는 archive 여부 무관 항상 접근 가능" 멘탈모델과 일치 — starred 처리한 메일에 새 답장이 와도 "새 메일"로 인지되길 원함. 단, 이 union은 `src/shared/view.ts`의 뷰 술어(Inbox/Starred는 이미 분리됨, `starred-view` D3)를 건드리지 않는 **알림 판정 전용 독립 쿼리**다.

## D3. 알림 내용 = 발신자+제목(1건) / 그룹화(2건+) — 사용자 확정
- **선택지**: (a) 개별 발신자+제목 미리보기, 2건+는 그룹화, (b) 항상 개수만("N개의 새 메일").
- **선택**: (a).
- **이유**: 1건일 때는 macOS Mail.app 관례대로 구체적 정보가 유용하고, 2건+를 전부 개별 알림으로 쏘면 60s 주기당 스팸처럼 도배될 수 있어 그룹화가 필요.

## D4. 클릭 동작 — 사용자 확정
- **선택지**: (a) 개별=스레드 오픈(필요시 계정 전환) / 그룹=활성 계정 Inbox로만 이동(계정 자동전환 없음), (b) 항상 앱 포커스만.
- **선택**: (a).
- **이유**: 개별 알림은 "그 메일"이 명확하니 바로 열어주는 게 직관적이고, 그룹 알림은 여러 계정에 걸쳐 있을 수 있어 사용자 모르게 계정이 전환되는 게 오히려 혼란스럽다 — 그룹일 때는 사용자가 있던 화면 그대로 유지.

## D5. 포커스 중 알림 억제 — 사용자 확정
- **선택지**: (a) 포커스 중이면 알림 안 띄움(배지는 갱신), (b) 포커스 여부 무관 항상 띄움.
- **선택**: (a).
- **이유**: 사용자가 이미 ZenMail을 보고 있으면 새 메일은 인박스에 바로 나타나므로 알림이 중복·소음이 된다 — macOS Mail/Slack 관례.

## D6. 설정 토글 없음, v1은 항상 ON — 사용자 확정
- **선택지**: (a) 토글 없이 항상 ON, (b) kbar에 on/off 토글 추가(settings KV persist).
- **선택**: (a).
- **이유**: YAGNI — 기존 다수 feature도 설정 없이 기본값으로 출하(예: select-all-in-view). 나중에 요청 있으면 `light-mode`의 토글 패턴(kbar+settings KV) 그대로 재사용 가능.

## D7. 새 폴링 루프를 만들지 않고 기존 60s 배지 데몬을 확장 — 추천안
- **컨텍스트**: `snooze.ts`의 60s 데몬이 이미 계정별 `inboxUnreadCount()`를 매 틱 호출해 사이드바 배지를 갱신 중(멀티 계정 D7 "실계정 API 비용 억제").
- **선택지**: (a) 이 배지 루프를 확장(카운트 증가 시에만 추가 조회), (b) 독립된 별도 폴링 타이머 신설.
- **선택**: (a).
- **이유**: (b)는 같은 데이터를 얻기 위해 실계정 Gmail API를 두 배로 호출하게 되어 D7 원칙과 정면 충돌한다. (a)는 "카운트가 늘었을 때만" 추가 1콜이라 평상시엔 비용이 그대로다.

## D8. 신규 unread 상세 조회는 카운트 증가(`n > ctx.unreadCount`) 시에만 — 추천안
- **컨텍스트**: 발신자/제목을 알림에 넣으려면 스레드 상세(제목/발신자 포함)를 가져오는 추가 API 콜이 필요.
- **선택지**: (a) `n > ctx.unreadCount`(순증)일 때만 추가 조회, (b) 매 틱 무조건 조회.
- **선택**: (a).
- **이유**: 매 틱 조회(b)는 사실상 배지 자체를 다시 폴링과 동급 비용으로 만든다 — 순증이 없으면(감소·동일) 어차피 새 메일이 없으므로 조회할 이유가 없다. 순증+순감소가 같은 틱에서 상쇄되는 극히 드문 경우는 알려진 한계로 문서화(PRD 논스코프).

## D9. 콜드스타트 baseline은 계정별 "최초 상세 조회" 시점에 시딩, `refreshBadges()`는 무변경 — 추천안
- **컨텍스트**: 로그인 직후 이미 있던 unread 메일이 전부 "새 메일"로 오폭발하면 안 됨(PRD R6).
- **선택지**: (a) `ctx.lastKnownUnreadIds === undefined`일 때(계정의 첫 상세 조회) baseline만 시딩하고 알림 0건, `refreshBadges()`는 카운트 시딩 역할 그대로 유지(무변경) — 이후 데몬 틱에서 알림 로직이 자연히 콜드스타트를 처리, (b) `refreshBadges()`에도 별도 콜드스타트 처리 로직 추가.
- **선택**: (a).
- **이유**: `refreshBadges()`가 이미 `ctx.unreadCount`를 로그인 직후 정확히 시딩해두므로, 데몬의 첫 틱에서 `n > ctx.unreadCount`가 열릴 일이 거의 없다(레이스가 있어도 그 좁은 창의 진짜 신규분만 대상이 됨). 두 곳에 콜드스타트 로직을 중복시키지 않고 한 곳(데몬의 `lastKnownUnreadIds` 최초-관측 가드)에만 두는 게 더 단순하고 버그 표면이 작다.

## D10. `lastKnownUnreadIds`는 모듈 전역 Map이 아니라 `AccountContext` 필드로 — 추천안
- **컨텍스트**: 계정별 "마지막으로 관측한 unread 스레드 ID 집합"을 어디에 보관할지.
- **선택지**: (a) `AccountContext`에 필드 추가(`unreadCount`처럼 이미 컨텍스트-온 상태 패턴), (b) `ipc.ts`/`notify.ts`에 별도 `Map<accountId, Set<string>>` 모듈 전역 변수 신설.
- **선택**: (a).
- **이유**: sign-out 시 `contexts.delete(email)`로 `AccountContext` 객체 자체가 버려지는데, (a)는 이 GC에 자동으로 올라타 별도 정리 코드가 필요 없다. (b)는 계정 삭제 시점마다 짝 맞춰 지워줘야 하는 새로운 실패 지점을 만든다.

## D11. 알림 판정용 union 쿼리는 `src/shared/view.ts`의 뷰 술어를 재사용하지 않고 독립 정의 — 추천안
- **컨텍스트**: `starred-view`가 Inbox/Starred 뷰를 의도적으로 분리(union 제거)했는데, 이번 feature는 "새 메일" 판정에서 그 둘을 다시 합쳐야 한다(D2).
- **선택지**: (a) `isInInboxView(labelIds, snoozeLabelId) || isInStarredView(labelIds, snoozeLabelId)`를 알림 로직에서만 조합해서 쓰고 `view.ts` 자체는 안 건드림, (b) `view.ts`에 세 번째 술어 `isInInboxOrStarredView`를 새로 만들어 export.
- **선택**: (a).
- **이유**: 이 union은 UI 뷰 개념이 아니라 "알림 판정" 전용 개념이라 `view.ts`(렌더러·캐시가 공유하는 UI 뷰 계약)에 넣으면 책임이 섞인다 — 이미 export된 두 술어를 notify.ts 내부에서 `||`로 조합하는 것으로 충분하고, 나중에 "Starred만 알림 끄기" 같은 요구가 생겨도 이 조합 지점만 고치면 된다.

## D12. 상세조회 실패 시에도 `ctx.unreadCount`는 갱신 — 최종 리뷰에서 확인된 트레이드오프 (수정 없이 문서화)
- **컨텍스트**: 최종 전체 브랜치 리뷰(deep-reasoner/Opus)에서 발견: `n > ctx.unreadCount`로 상세조회(listThreads)를 시도했는데 그 콜 자체가 실패하면(네트워크 등, catch에서 로그만 남김), `ctx.lastKnownUnreadIds`는 갱신되지 않지만 `ctx.unreadCount = n`은 try/catch 바깥이라 무조건 실행된다.
- **선택지**: (a) 현재 그대로 — 배지는 항상 정확한 값을 보여주고, 놓친 신규분은 `lastKnownUnreadIds`가 갱신 안 됐으므로 다음에 카운트가 그 수준을 다시 넘어설 때 자연 복구, (b) 상세조회 성공 시에만 `ctx.unreadCount`도 갱신(배지가 실패한 틱 동안 실제보다 낮게 표시될 수 있음).
- **선택**: (a), 수정 없이 유지.
- **이유**: 배지 정확성이 알림의 베스트-에포트 신뢰성보다 사용자에게 더 중요하다고 판단(배지는 사이드바 unreadCount와도 공유되는 값이라 (b)는 두 표시가 서로 다른 시점 값을 갖게 되는 부작용도 생긴다). 완전 손실이 아니라 "그 사이에 해당 스레드들이 읽음 처리되는" 좁은 경우에만 알림이 안 뜨고 배지는 항상 맞다 — Low 심각도로 판단해 수정하지 않되, 인라인 주석(`snooze.ts`)을 "다음 틱에 재시도"라는 부정확한 표현에서 실제 복구 조건을 설명하도록 정정했다.

## D13. 알림 클릭 시 창이 완전히 종료된 상태(macOS)의 재오픈은 이번 feature 범위 밖 — 최종 리뷰에서 발견, 백로그 이월
- **컨텍스트**: macOS에서 사용자가 창을 완전히 닫으면(앱은 Dock에 남고 데몬은 계속 돎) `getWindow()`가 `null`을 반환 — 이 상태에서 새 메일 알림은 정상 발화되지만, 그 알림을 클릭해도 `getWindow()?.show()`가 no-op이라 창이 다시 열리지 않는다(크래시는 아님, 단순히 아무 일도 안 일어남). 창이 백그라운드에 살아있는 일반적인 "다른 앱 포커스 중" 상태는 `show()`가 정상 동작해 영향 없음.
- **선택지**: (a) 이번 feature 범위 밖으로 두고 백로그 이월, (b) `notify.ts`에 `createWindow` 팩토리를 주입받아 클릭 시 창이 없으면 재생성.
- **선택**: (a).
- **이유**: 발생 빈도가 낮은 edge case(완전 종료 후에도 데몬이 계속 돌고 있어야 함)이고, 고치려면 `notify.ts`가 main 프로세스의 윈도우 생성 로직까지 알아야 해 책임 경계가 커진다 — 다음 세션에서 원할 때 별도로 처리.
